# Dissertation Test Stand: Monolith vs Microservices

## Описание проекта

Данный репозиторий содержит программный испытательный стенд, разработанный в рамках диссертационного исследования. Главная цель проекта — эмпирическое сравнение производительности, потребления системных ресурсов (RAM/CPU) и сетевых задержек (latency) между монолитной и микросервисной архитектурами при одинаковой бизнес-логике.

Бизнес-кейс имитирует высоконагруженную систему обработки финансовых транзакций (депозитов), включающую API-шлюз, тяжелую CPU-bound валидацию и I/O-bound работу с базой данных.

---

## Архитектура и принцип работы

В проекте реализованы две независимые архитектурные модели:

1. **Monolith (Монолит):** Вся логика (маршрутизация, валидация, транзакции) выполняется внутри единого процесса и адресного пространства. Вызовы функций происходят в оперативной памяти.

2. **Microservices (Микросервисы):** Логика декомпозирована на три независимых сервиса, общающихся по протоколу HTTP/JSON внутри виртуальной сети Docker:
   - **API Gateway** — принимает внешние запросы и маршрутизирует трафик.
   - **Validator Service** — имитирует тяжелые вычислительные задачи (CPU-bound). Поддерживает горизонтальное масштабирование от 1x до 3x реплик.
   - **Transaction Service** — отвечает за запись данных (I/O-bound).

## Требования для запуска

Для локального запуска стенда необходимы:

- **Docker** и **Docker Compose**
- **k6** (для запуска нагрузочных тестов)
- **Go 1.21+** (опционально, для локальной разработки)

---

## Инструкция по запуску

### Сценарий 1: Запуск монолита

1. Поднимите инфраструктуру и приложение:

```bash
docker-compose -f docker-compose.mono.yml up --build -d
```

2. Убедитесь, что контейнер работает и утилизирует ресурсы:

```bash
docker stats
```

3. Запустите нагрузочный тест:

```bash
k6 run k6/load_test.js
```

4. Остановите среду:

```bash
docker-compose -f docker-compose.mono.yml down
```

---

### Сценарий 2: Запуск микросервисов (масштабированная среда)

1. Поднимите микросервисную среду с тремя репликами валидатора:

```bash
docker-compose up --build --scale validator=3 -d
```

2. Проверьте распределение ресурсов между контейнерами:

```bash
docker stats
```

3. Запустите нагрузочный тест:

```bash
k6 run k6/load_test.js
```

4. Остановите среду:

```bash
docker-compose down
```

## Результаты исследования

Анализ метрик, полученных с помощью данного стенда, подтверждает наличие **инфраструктурного налога** в распределенных системах:

| Метрика | Результат |
|---|---|
| **RAM** | Декомпозиция увеличивает потребление оперативной памяти более чем на 300% |
| **Latency** | Максимальное время отклика возрастает на 67% из-за затрат на сериализацию JSON и сетевой джиттер |
| **Масштабирование** | Горизонтальное масштабирование CPU-bound компонентов эффективно только при достижении предела вычислительных ресурсов единого узла |

Подробные выводы, графики и обоснование концепции **точки перегиба (Tipping Point)** представлены в тексте диссертационной работы.
